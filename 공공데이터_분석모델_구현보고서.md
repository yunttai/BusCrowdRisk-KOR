# 공공데이터 분석 모델 구현 보고서

이 문서는 공공데이터 공모전 보고서에 넣기 위해, 현재 프로젝트에서 실제로 구현한 데이터 수집, 전처리, feature 설계, 학습, 검증, 서비스 연동 과정을 정리한 문서다.

현재 기준 핵심 결론:

```text
최종 만차확률은 LightGBM 기반 모델 artifact로 계산한다.
실시간 차량 정보가 있으면 실시간 row를 사용하고,
실시간 차량 정보가 없으면 시간표/과거이력 기반 schedule_prior proxy row를 만들어 같은 모델에 입력한다.
```

현재 주요 산출물:

```text
data/risk_model_lightgbm.pkl
data/risk_model_lightgbm_artifact.json
data/model_diagnostics.json
data/model_diagnostics.md
data/schedule_prior_backtest.json
data/schedule_prior_backtest.md
data/training_dataset.csv
data/model_hourly_features.csv
```

# 1. 공공데이터 수집 구조 설계 및 구현

## 1.1 수집 대상

이 프로젝트는 경기도 용인 5000번대 광역버스 계열의 만차 위험도를 예측하기 위해 아래 공공데이터를 중심으로 수집 구조를 설계했다.

| 구분 | 데이터 | 역할 |
| --- | --- | --- |
| 버스 실시간 | 경기도 버스 위치정보 | 차량별 현재 위치, 정류소 순번, 잔여좌석, 혼잡도 수집 |
| 버스 도착 | 경기도 버스 도착정보 | 사용자가 탈 정류소 기준 도착 예정 차량 확인 |
| 노선 정보 | 경기도 버스노선 조회 | 버스번호를 `routeId`로 변환 |
| 정류소 정보 | 경기도 정류소 조회 | 정류소명을 `stationId`로 변환 |
| 경유 정류소 | 노선별 경유정류소 | 노선 내 정류소 순번과 방향 파악 |
| 날씨 | 기상청 ASOS 시간자료 | 시간대별 기온, 강수, 습도, 풍속 등 외부요인 |
| 특일 | 한국천문연구원 특일 정보 | 공휴일, 대체공휴일, 특일 feature 생성 |
| 행사 | 행사/관광/문화행사 데이터 | 날짜별 행사 영향 feature 생성 |

현재 실제 적재된 데이터 규모:

```text
bus_location_snapshot: 87,887건
preprocessed_location_features: 87,887건
model_hourly_features: 87,887건
model_target_labels: 87,887건
external_weather_hourly: 274건
external_holiday_daily: 22건
external_event_daily: 12건
```

현재 학습 feature 기준 데이터 범위:

```text
2026-05-13 01:18:43 KST
~
2026-05-24 19:03:33 KST
```

## 1.3 대상 노선

현재 주요 분석 대상은 용인 5000번대 광역버스 계열이다.

```text
5000
5001A/B
5001-1A/B
5003A/B
5005
```

노선은 사용자가 입력하는 버스번호가 아니라 실제 API에서 쓰는 `routeId` 기준으로 관리한다.  
예를 들어 `5001-1B`는 현재 DB에서 다음 값으로 관리된다.

```text
routeId: 228000177

```

# 2. 수집 데이터 정제 및 전처리

## 2.1 원본 데이터 import

확보한 실측 백업 데이터:

```text
sureing_occupancy_backup.sql
```

이 dump 파일을 SQLite 학습 파이프라인에 적재했다.

처리 원칙:

```text
remaining_seat_count = -1
  -> remain_seat_cnt = 0
  -> 만차로 학습
```

이유:

```text
해당 데이터에서 -1은 실제 결측이 아니라 잔여좌석 0, 즉 만차 상태로 해석하기로 했다.
```

관련 명령:

```powershell
busseat import-sureing-dump .\sureing_occupancy_backup.sql
```

## 2.2 시간대 전처리

수집 시각은 KST 기준으로 정리했다.

생성한 시간 feature:

```text
service_date
day_of_week
day_name_ko
is_weekend
is_holiday
holiday_name
hour
time_bucket_10m
time_period
observed_hour_kst
```

이 처리를 통해 같은 노선이라도 출근, 퇴근, 야간, 공휴일 여부에 따라 다른 패턴을 학습할 수 있게 했다.

## 2.3 정류소/노선 정규화

노선명은 모델 입력에서 흔들리지 않도록 canonical 형태로 정규화했다.

예:

```text
5001-1B
5001-1A
5005
```

정류소는 `stationId`, `stationSeq`, `stationName`을 함께 저장했다.

```text
stationId: 정류소 고유 ID
stationSeq: 노선 내 정류소 순번
stationName: 표시용 정류소명
```

예:

```text
5001-1B 신논현역
stationId: 122000616
stationSeq: 43
```

## 2.4 결측 처리 원칙

외부요인은 결측을 무조건 채워 넣지 않고, 결측 여부를 feature로 남긴다.

예:

```text
weather_imputed
air_quality_imputed
traffic_imputed
event_imputed
```

현재 모델에 포함한 외부요인:

```text
weather
event
```

현재 제외한 외부요인:

```text
expected_boardings
air_quality
traffic
```

제외 이유:

```text
현재 적재 데이터가 부족하거나 결측 보정률이 높아, 그대로 넣으면 가짜 상관관계를 만들 수 있기 때문이다.
```

# 3. 분석 목적에 맞는 Feature 설계

## 3.1 Feature 설계 목표

목표는 단순히 현재 좌석이 있는지 보는 것이 아니라, 다음 정류소 기준으로 만차가 될 위험을 예측하는 것이다.

따라서 feature는 다음 질문에 답하도록 설계했다.

```text
1. 지금 차량 좌석 상태가 어떤가?
2. 어느 노선인가?
3. 노선의 어느 위치인가?
4. 몇 시, 무슨 요일, 공휴일인가?
5. 날씨나 행사가 수요에 영향을 줄 수 있는가?
```

## 3.2 버스 상태 feature

```text
remain_seat_cnt
crowded
crowded_label
estimated_capacity
seat_scarcity_score
is_no_seat
is_low_seat_2
is_low_seat_5
```

`remain_seat_cnt`는 가장 중요한 실측값이다.  
다만 잔여좌석 수는 이산 구간으로 자르지 않고 로그 기반 스코어로도 변환했다.

```text
seatScarcityScore
  = 100 * (1 - (ln(remainSeatCnt + 1) / ln(seatCapacity + 1)) ^ 2.15)
```

주의:

```text
seatScarcityScore는 최종 확률이 아니다.
모델 입력 feature이자 화면 표시용 보조 점수다.
```

## 3.3 노선 feature

```text
route_id
route_name
canonical_route_name
route_type_cd
route_type_name
```

노선별로 승객 패턴, 운행 구간, 만차 발생 구간이 다르기 때문에 노선 feature를 포함했다.

## 3.4 정류소 순번 feature

```text
station_id
station_seq
station_name
station_seq_segment
```

`station_seq`는 노선 내 위치를 나타낸다.  
같은 시간대라도 노선 초반과 후반은 만차 위험이 다르므로 중요한 feature다.

`station_seq_segment`는 순번을 구간화한 해석용 값이다.

```text
origin_0_25
early_mid_25_50
late_mid_50_75
terminal_75_100
```

## 3.5 시간 feature

```text
hour
time_bucket_10m
time_period
day_of_week
is_weekend
is_holiday
holiday_name
```

출근/퇴근/주말/공휴일 패턴을 반영하기 위한 feature다.

## 3.6 외부요인 feature

현재 모델에 포함한 외부요인:

```text
temperature
precipitation
humidity
wind_speed
cloud_amount
weather_text
weather_imputed
event_count
event_nearby_count
event_imputed
```

중요 원칙:

```text
날씨 +8점, 교통정체 +10점 같은 수동 가중치를 쓰지 않는다.
외부요인은 원자료 feature로 넣고, 모델이 학습하게 한다.
```

# 5. 예측/분석 모델 구성

## 5.1 최종 모델

현재 최종 예측 모델은 LightGBM classifier다.

```text
modelName: lightgbm_lgbm_classifier
modelVersion: lightgbm_full_risk_v1
artifact: data/risk_model_lightgbm.pkl
threshold: 0.9
```

모델 입력 feature 그룹:

```text
bus_state
route
station_seq
time
weather
event
```

제외 feature 그룹:

```text
expected_boardings
air_quality
traffic
```

제외 이유:

```text
현재 원자료가 부족하거나 결측 보정률이 높아 모델 오염 가능성이 있기 때문이다.
```

## 5.2 모델 선택 이유

LightGBM을 사용한 이유:

```text
1. 비선형 패턴 학습 가능
2. 결합 feature가 많은 tabular data에 적합
3. 노선, 정류소 순번, 시간대, 날씨 feature 간 상호작용 학습 가능
4. 학습과 추론 속도가 빠름
5. feature importance 기반 설명 가능성 확보 가능
```

이 프로젝트는 이미지나 자연어가 아니라 정형 데이터 기반 예측 문제이므로, 딥러닝보다 Gradient Boosting 계열 모델이 더 적합하다.

## 5.3 예측 모드

현재 예측 모드는 두 가지다.

### live_vehicle

```text
GBIS 실시간 도착/위치정보에서 실제 차량 row를 만들 수 있을 때 사용
```

흐름:

```text
도착정보 조회
-> 위치정보 조회
-> 차량번호/vehId로 매칭
-> 실시간 feature row 생성
-> LightGBM predict_proba
```

### schedule_prior

```text
실시간 차량 row를 만들 수 없을 때 사용
```

흐름:

```text
입력 시간 확인
-> 공휴일/평일 판단
-> 시간표 운행 여부 확인
-> 과거 이력에서 proxy 잔여좌석/혼잡도 산출
-> 같은 LightGBM 모델에 입력
```

중요:

```text
schedule_prior는 다른 모델이 아니다.
입력 row만 실시간 차량 대신 시간표/과거이력 proxy로 만든다.
```

## 5.4 전체 모델 파이프라인

현재 모델 파이프라인은 다음 순서로 동작한다.

```text
원본 수집 데이터
-> bus_location_snapshot 적재
-> preprocessed_location_features 생성
-> model_hourly_features 생성
-> model_target_labels 생성
-> 학습 가능한 row만 추출
-> FeatureSelector로 사용할 feature 그룹 결정
-> FeatureEncoder로 숫자형/범주형 feature 벡터화
-> LightGBM binary classifier 학습
-> data/risk_model_lightgbm.pkl artifact 저장
-> 실시간 또는 schedule_prior 입력 row 생성
-> artifact 내부 encoder로 같은 방식 변환
-> model.predict_proba()로 만차확률 산출
-> JSON/API 응답용 결과 가공
```

핵심은 학습과 예측이 같은 전처리 객체를 공유한다는 점이다.

```text
학습 시점:
  selector.fit 기준 결정
  encoder.fit 기준 결정
  model 학습
  selector + encoder + model을 artifact에 함께 저장

예측 시점:
  artifact 로드
  artifact.encoder.transform(row)
  artifact.model.predict_proba(row)
```

따라서 운영 시 새 row가 들어와도 학습 때와 동일한 feature 순서, 평균/표준편차, 범주형 one-hot 기준을 사용한다.

## 5.5 FeatureSelector 구조

`FeatureSelector`는 모든 데이터를 무조건 모델에 넣지 않는다.  
결측이 심한 외부요인은 모델을 오염시킬 수 있기 때문에, feature 그룹별로 포함 여부를 자동 판단한다.

항상 기본 후보로 보는 그룹:

```text
bus_state
route
station_seq
time
```

외부요인 포함 기준:

| Feature group | 포함 기준 | 현재 상태 |
| --- | --- | --- |
| expected_boardings | 결측률 50% 이하 | 제외 |
| weather | `weather_imputed` 비율 30% 이하 | 포함 |
| air_quality | `air_quality_imputed` 비율 50% 이하 | 제외 |
| traffic | `traffic_imputed` 비율 50% 이하 | 제외 |
| event | `event_imputed` 비율 80% 이하 | 포함 |

현재 최종 artifact에 포함된 feature group:

```text
bus_state
route
station_seq
time
weather
event
```

현재 최종 artifact에서 제외된 feature group:

```text
expected_boardings
air_quality
traffic
```

이렇게 한 이유는 단순하다.

```text
외부요인을 많이 넣는 것보다,
신뢰 가능한 외부요인만 넣는 것이 실제 예측 안정성에 더 중요하다.
```

## 5.6 FeatureEncoder 구조

`FeatureEncoder`는 모델 입력을 숫자 벡터로 바꾼다.

숫자형 feature는 평균과 표준편차로 표준화한다.

```text
encoded_value = (value - train_mean) / train_std
```

범주형 feature는 one-hot encoding으로 변환한다.

예:

```text
canonical_route_name=5005
canonical_route_name=5001-1B
time_period=commute_evening
crowded_label=혼잡
weather_text=비
```

현재 사용하는 주요 숫자형 feature:

| 그룹 | Feature |
| --- | --- |
| bus_state | `remain_seat_cnt`, `seat_scarcity_score`, `crowded`, `estimated_capacity`, `is_no_seat`, `is_low_seat_2`, `is_low_seat_5` |
| station_seq | `station_seq` |
| time | `hour_sin`, `hour_cos`, `day_sin`, `day_cos`, `is_weekend`, `is_holiday` |
| weather | `temperature`, `precipitation`, `humidity`, `wind_speed`, `cloud_amount`, `weather_imputed` |
| event | `event_count`, `event_nearby_count`, `event_imputed` |

현재 사용하는 주요 범주형 feature:

| 그룹 | Feature |
| --- | --- |
| route | `canonical_route_name` |
| time | `time_period` |
| bus_state | `crowded_label` |
| weather | `weather_text` |

시간 feature에서 `hour_sin`, `hour_cos`, `day_sin`, `day_cos`를 쓰는 이유:

```text
23시와 0시는 숫자로는 멀어 보이지만 실제 시간 흐름에서는 가깝다.
월요일과 일요일도 단순 숫자 차이만으로 처리하면 주기성이 깨진다.
따라서 시간/요일은 sin/cos 주기 feature로 변환해 모델이 순환 구조를 학습하게 했다.
```

## 5.7 LightGBM 학습 구조

최종 모델은 이진 분류 모델이다.

```text
입력: 차량/노선/정류소/시간/날씨/행사 feature
출력: 다음 정류소 기준 만차일 확률
target: target_no_seat_next_station
```

학습 target의 의미:

```text
현재 row의 차량이 다음 정류소에서 remain_seat_cnt == 0 이 되는가
```

모델 학습 시 사용하는 주요 설정:

| 항목 | 값 |
| --- | ---: |
| Model | `LGBMClassifier` |
| Objective | binary |
| n_estimators | 220 |
| learning_rate | 0.045 |
| num_leaves | 31 |
| max_depth | -1 |
| subsample | 0.9 |
| colsample_bytree | 0.9 |
| reg_lambda | 0.1 |
| random_state | 20260524 |

클래스 불균형 보정도 적용했다.

현재 만차 target의 positive rate는 약 12.7%라서, 그냥 학습하면 모델이 “안 만차” 쪽으로 치우칠 수 있다.  
그래서 학습 sample weight를 다음 방식으로 준다.

```text
positive_weight = 전체 row 수 / (2 * positive row 수)
negative_weight = 전체 row 수 / (2 * negative row 수)
```

즉 만차 row가 적기 때문에 만차 row의 학습 가중치를 더 크게 잡는다.

## 5.8 모델 artifact 구성

최종 artifact는 모델 파일 하나에 예측에 필요한 객체를 함께 저장한다.

```text
data/risk_model_lightgbm.pkl
```

artifact 내부 구성:

| 항목 | 의미 |
| --- | --- |
| `version` | 모델 버전 |
| `targetColumn` | 학습 target |
| `threshold` | high risk 판정 기준 |
| `modelName` | 사용 모델 이름 |
| `model` | 학습된 LightGBM 모델 |
| `selector` | feature 그룹 선택 기준 |
| `encoder` | 숫자형 표준화/범주형 one-hot 기준 |
| `featureNames` | 최종 feature 벡터 순서 |
| `includedFeatureGroups` | 포함된 feature 그룹 |
| `excludedFeatureGroups` | 제외된 feature 그룹 |
| `trainingRows` | 학습 row 수 |
| `positiveRate` | target 양성 비율 |
| `maxStationSeqByRoute` | 노선별 최대 정류소 순번 |

이 구조 덕분에 서비스 서버는 학습 CSV 전체를 다시 읽지 않아도 된다.  
예측 시에는 artifact와 현재 입력 row만 있으면 된다.

## 5.9 예측 입력 row 생성 방식

예측은 항상 같은 LightGBM artifact로 수행한다.  
달라지는 것은 모델이 아니라 입력 row를 만드는 방식이다.

### 실시간 row

실시간 차량 정보가 있을 때:

```text
GBIS 도착정보
-> 도착 예정 차량 확인
-> GBIS 위치정보
-> 차량번호/차량 ID 매칭
-> 현재 remainSeatCnt, crowded, stationSeq 확보
-> 외부요인 feature 결합
-> LightGBM 예측
```

이 경우는 실제 차량 상태가 들어가므로 가장 신뢰도가 높다.

### schedule_prior proxy row

실시간 차량 정보가 없을 때:

```text
사용자 입력 시각
-> 공휴일/평일/주말 판단
-> 노선 시간표 기준 운행 가능 여부 확인
-> 과거 이력에서 같은 조건의 잔여좌석 중앙값 산출
-> 과거 이력에서 혼잡도 최빈값 산출
-> proxy remainSeatCnt/crowded 생성
-> 같은 LightGBM artifact에 입력
```

이 방식은 “해당 시각에 실제 차량이 API에 없을 때”도 예측 결과를 낼 수 있게 해준다.  
단, 실시간 관측이 아니라 과거 패턴 기반이므로 결과에 `historicalProxyGroup`, `historicalProxyRows`를 반드시 같이 표시한다.

## 5.10 schedule_prior fallback 순서

schedule_prior는 과거 이력을 아무렇게나 평균내지 않는다.  
가장 정확한 조건부터 찾고, 없을 때만 점점 넓은 조건으로 fallback한다.

현재 fallback 순서:

| 순서 | 그룹 | 의미 |
| ---: | --- | --- |
| 1 | `route_station_hour_daytype` | 같은 노선 + 같은 정류소 + 같은 시간 + 같은 주말/공휴일 조건 |
| 2 | `route_station_adjacent_hour_daytype` | 같은 노선 + 같은 정류소 + 인접 시간대 + 같은 주말/공휴일 조건 |
| 3 | `route_station_hour` | 같은 노선 + 같은 정류소 + 같은 시간 |
| 4 | `route_station_daytype` | 같은 노선 + 같은 정류소 + 같은 주말/공휴일 조건 |
| 5 | `route_station` | 같은 노선 + 같은 정류소 |
| 6 | `route_station_seq_hour` | 같은 노선 + 같은 정류소 순번 + 같은 시간 |
| 7 | `route_hour` | 같은 노선 + 같은 시간 |
| 8 | `route` | 같은 노선 |
| 9 | `hour` | 같은 시간 |

신뢰도 기준:

```text
route_station_hour_daytype
route_station_adjacent_hour_daytype
  -> 상대적으로 신뢰 가능

route_hour 이하
  -> 정류소 맥락이 약해져 신뢰도 낮음
```

예측 JSON에서 반드시 봐야 하는 값:

```text
predictionMode
historicalProxyGroup
historicalProxyRows
proxyRemainSeatCnt
proxyCrowded
historicalNextStationFullRate
historicalNowFullRate
```

## 5.11 최종 확률 계산 방식

최종 만차확률은 사람이 정한 수식으로 계산하지 않는다.

사용하지 않는 방식:

```text
잔여좌석 점수 + 날씨 가중치 + 공휴일 가중치 + 행사 가중치
```

현재 사용하는 방식:

```text
feature row
-> artifact.encoder.transform(row)
-> artifact.model.predict_proba(encoded_row)
-> fullSeatProbability
-> fullSeatRiskScore = fullSeatProbability * 100
```

즉 `remainSeatCnt`, `weather`, `holiday`, `event` 등이 확률에 영향을 주긴 하지만, 그 영향력은 사람이 임의로 더하는 것이 아니라 LightGBM이 학습 데이터에서 자동으로 학습한다.

출력값 해석:

| 출력 | 의미 |
| --- | --- |
| `fullSeatProbability` | 모델이 계산한 다음 정류소 만차 확률 |
| `fullSeatRiskScore` | 확률을 0~100 점수로 표시한 값 |
| `isFullRisk` | threshold 이상 여부 |
| `riskLevel` | low/medium/high 표시용 등급 |

## 6.5 검증 파이프라인 구분

이 프로젝트의 검증은 하나로 끝내면 안 된다.  
예측 방식이 두 가지이기 때문이다.

```text
1. 실시간 차량 상태 feature가 있는 모델 검증
2. 실시간 차량 상태가 없을 때의 schedule_prior 검증
```

### 6.5.1 실시간 feature 기반 검증

이 검증은 모델 자체의 분류 성능을 확인한다.

검증 구조:

```text
전체 학습 row 시간순 정렬
-> 앞 80% train
-> 뒤 20% test
-> train으로 FeatureSelector/Encoder/LightGBM 학습
-> test row의 실제 feature를 그대로 넣어 예측
-> 실제 target_no_seat_next_station과 비교
```

이 검증에서 의미하는 것:

```text
실시간 API에서 차량의 remainSeatCnt/crowded/stationSeq를 가져올 수 있으면
모델이 다음 정류소 만차 여부를 얼마나 잘 맞추는가
```

따라서 이 결과는 “실시간 차량 row가 있는 상황”의 성능이다.

### 6.5.2 schedule_prior 검증

이 검증은 실시간 차량 row가 없는 상황을 따로 검증한다.

검증 구조:

```text
전체 학습 row 시간순 정렬
-> 앞 80% train
-> 뒤 20% test
-> test row의 실제 remainSeatCnt/crowded를 숨김
-> train 이력만으로 proxy remainSeatCnt/crowded 생성
-> 같은 LightGBM 모델에 proxy row 입력
-> 숨겨둔 실제 target과 비교
```

이 검증에서 의미하는 것:

```text
지금 실시간 차량이 API에 없더라도,
시간표와 과거 이력만으로 만든 proxy row가 실제 만차 위험을 어느 정도 설명하는가
```

따라서 schedule_prior 결과는 모델 성능뿐 아니라 과거 이력 매칭 품질의 영향을 받는다.

## 6.6 예측 신뢰도 판단 기준

서비스나 보고서에서 예측값을 해석할 때는 확률만 보면 안 된다.  
특히 `schedule_prior`에서는 아래 항목을 같이 봐야 한다.

| 항목 | 해석 |
| --- | --- |
| `predictionMode` | `live_vehicle`이면 실시간, `schedule_prior`이면 과거이력 기반 |
| `historicalProxyGroup` | 어떤 조건으로 과거 이력을 찾았는지 |
| `historicalProxyRows` | 참조한 과거 row 수 |
| `stationSeq` | 정류소 순번이 정상 매칭됐는지 |
| `stationName` | 정류소명이 `unknown`이 아닌지 |
| `scheduledOperating` | 해당 시각 운행 가능 시간대인지 |

권장 해석:

```text
predictionMode = live_vehicle
  -> 가장 신뢰 가능

historicalProxyGroup = route_station_hour_daytype
historicalProxyGroup = route_station_adjacent_hour_daytype
  -> schedule_prior 중 상대적으로 신뢰 가능

historicalProxyGroup = route_hour 이하
stationName = unknown
stationSeq = 0
  -> 정류소 맥락이 약하므로 서비스에서 신뢰도 낮음으로 표시
```

## 6.7 모델 결과가 이상해 보일 때 점검 순서

예측 결과가 상식과 다르게 보이면 아래 순서로 확인한다.

```text
1. stationId가 해당 노선의 실제 정류소인지 확인
2. routeName과 routeId가 맞는지 확인
3. stationName이 unknown인지 확인
4. stationSeq가 0인지 확인
5. predictionMode가 live_vehicle인지 schedule_prior인지 확인
6. schedule_prior라면 historicalProxyGroup 확인
7. historicalProxyRows가 너무 적은지 확인
8. 공휴일/평일 판정이 맞는지 확인
9. scheduleDayType이 weekday/saturday/sunday_holiday 중 맞게 들어갔는지 확인
10. 같은 조건의 실제 DB row를 조회해 remainSeatCnt 분포를 확인
```

예를 들어 `5001-1B 신논현역`은 `stationId=122000616`이 맞다.  
`stationId=121000944`를 넣으면 다른 방향/다른 정류소로 처리되어 `stationName=unknown`, `stationSeq=0`이 될 수 있고, 이 경우 결과를 신뢰하면 안 된다.

# 7. 평가지표 기반 성능 분석

## 7.1 사용한 평가지표

| 지표 | 의미 |
| --- | --- |
| Accuracy | 전체 예측 중 맞은 비율 |
| Precision | 만차라고 예측한 것 중 실제 만차 비율 |
| Recall | 실제 만차 중 모델이 잡아낸 비율 |
| F1 | Precision과 Recall의 조화 평균 |
| ROC-AUC | threshold와 무관한 분류 구분력 |
| Brier score | 확률 예측의 평균 제곱 오차 |
| Top decile precision | 위험도 상위 10%의 실제 만차 비율 |
| Calibration ECE | 예측 확률과 실제 발생률의 차이 |

## 7.2 왜 정확도만 보지 않는가

현재 target의 positive rate는 약 12.7%다.

```text
만차가 아닌 경우가 훨씬 많다.
```

이런 불균형 데이터에서는 모델이 대부분을 “안 만차”로 예측해도 accuracy가 높게 나올 수 있다.  
따라서 precision, recall, F1, ROC-AUC, Brier score를 함께 봐야 한다.

## 7.3 threshold 해석

현재 high risk threshold:

```text
0.9
```

의미:

```text
fullSeatProbability >= 0.9 이면 high risk
```

운영 해석:

```text
threshold 0.9는 거짓 경고를 줄이는 보수적 기준이다.
만차를 더 많이 잡고 싶으면 0.7~0.8 threshold 실험이 필요하다.
```

## 7.4 실시간 예측과 schedule_prior 예측의 차이

실시간 차량 row 기반 예측:

```text
현재 차량의 실제 remainSeatCnt, crowded, stationSeq를 사용할 수 있음
성능이 높음
```

schedule_prior 예측:

```text
실시간 차량 row가 없을 때 과거 이력으로 proxy row를 만듦
fallback 단계에 따라 신뢰도 차이가 큼
```

따라서 서비스와 보고서에서는 두 예측을 분리해서 표현해야 한다.

## 8.4 서비스 표시 기준

서비스 화면에서는 확률만 보여주면 안 된다.  
특히 `schedule_prior`는 과거 이력 기반이므로 신뢰도 정보를 함께 보여줘야 한다.

권장 표시:

```text
만차확률: 92.81%
위험도: 높음
예측방식: 시간표/과거이력 기반
참조 이력: 같은 노선/정류소/평일/인접시간대 11건
주의: 실시간 차량 정보가 없어 과거 패턴으로 예측한 값
```

## 8.5 사용하면 안 되는 결과

아래 경우는 보고서나 서비스 핵심 예측으로 쓰면 안 된다.

```text
stationName = unknown
stationSeq = 0
stationSeqSegment = unknown
historicalProxyGroup = route_hour 이하
historicalProxyRows가 매우 적음
```

이런 경우는 정류소 매칭 실패 또는 낮은 단계 fallback으로 판단하고, 사용자에게 신뢰도 낮음을 표시해야 한다.

# 최종 정리

현재 프로젝트는 다음 8개 항목을 실제 구현 범위로 설명할 수 있다.

```text
1. 공공데이터 수집 구조 설계 및 구현
2. 수집 데이터 정제 및 전처리
3. 분석 목적에 맞는 Feature 설계
4. 학습 데이터셋 생성
5. 예측/분석 모델 구성
6. 모델 학습 및 결과 검증
7. 평가지표 기반 성능 분석
8. 서비스 연동을 위한 예측 결과 가공
```

단, 보고서에서 반드시 구분해야 하는 부분:

```text
실시간 차량 row 기반 예측은 모델 성능이 높다.
schedule_prior는 실시간 차량이 없을 때의 보조 예측이며, fallback 단계와 표본 수에 따라 신뢰도가 달라진다.
```

따라서 최종 발표 문장은 다음처럼 쓰는 것이 안전하다.

```text
본 프로젝트는 경기도 버스 위치/도착 정보와 날씨, 공휴일, 행사 등 외부요인을 결합해
다음 정류소 기준 만차 위험도를 예측하는 LightGBM 기반 모델을 구현하였다.
시간순 holdout 검증에서 실시간 차량 feature 기반 모델은 ROC-AUC 0.9767, F1 0.9175를 기록했으며,
실시간 차량 정보가 없는 상황을 위한 schedule_prior backtest도 별도로 수행해
시간표/과거이력 기반 예측의 신뢰도와 한계를 분리 검증하였다.
```
