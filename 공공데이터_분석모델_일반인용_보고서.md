# 공공데이터 기반 버스 만차위험도 분석 모델 설명서

이 문서는 AI나 데이터 분석을 잘 모르는 사람도 이해할 수 있도록, 이 프로젝트에서 데이터를 어떻게 정리하고 어떤 방식으로 만차위험도를 예측했는지 핵심 개념만 정리한 문서다.

---

# 1. 프로젝트 목표

이 프로젝트의 목표는 사용자가 특정 시간에 특정 정류장에서 버스를 탈 때, 그 버스가 만차일 가능성이 높은지 미리 알려주는 것이다.

단순히 현재 좌석 수만 보여주는 것이 아니라, 아래 정보를 함께 사용해서 다음 정류소 기준 만차위험도를 계산했다.

```text
버스 잔여좌석
버스 혼잡도
노선
정류소 위치
요일과 시간대
공휴일 여부
날씨
행사 정보
과거 같은 조건의 만차 패턴
```

핵심 아이디어:

```text
비슷한 노선, 비슷한 정류소, 비슷한 시간대, 비슷한 날씨에서
과거에 자주 만차가 됐다면 앞으로도 만차위험이 높을 수 있다.
```

---

# 2. 데이터 전처리

## 2.1 왜 전처리가 필요한가

공공데이터 API에서 받은 데이터는 그대로 모델에 넣기 어렵다.

예를 들어 같은 버스라도 데이터마다 표현 방식이 다를 수 있다.

```text
5001-1
5001-1A
5001-1B
```

정류소도 이름만으로는 헷갈릴 수 있다.

```text
신논현역
신논현역.주류성빌딩
신논현역 정류소 ID
```

그래서 먼저 데이터를 예측에 사용할 수 있는 형태로 정리했다.

## 2.2 버스 데이터 정리

버스 위치 데이터에서 필요한 값만 골라 정리했다.

```text
노선 ID
노선명
정류소 ID
정류소명
정류소 순번
차량번호
잔여좌석 수
혼잡도
수집 시각
```

정리한 이유:

```text
같은 노선이라도 어느 정류소에 있느냐에 따라 만차위험이 달라지기 때문이다.
```

예를 들어 노선 초반에서는 좌석이 남아 있을 가능성이 크고, 주요 탑승 구간을 지난 뒤에는 만차가 될 가능성이 커질 수 있다.

## 2.3 시간 데이터 정리

수집 시각을 그대로 쓰지 않고, 분석하기 쉬운 시간 정보로 바꿨다.

```text
날짜
요일
시간
출근 시간대
퇴근 시간대
주말 여부
공휴일 여부
```

정리한 이유:

```text
월요일 오전 8시와 일요일 새벽 4시는 버스 수요가 완전히 다르기 때문이다.
```

## 2.4 잔여좌석 데이터 정리

잔여좌석 수는 이 프로젝트에서 가장 중요한 데이터다.

처리 기준:

```text
잔여좌석 0석 -> 만차
잔여좌석 -1 -> 실제 데이터 기준 만차로 해석
잔여좌석이 적을수록 위험도가 커지는 정보로 사용
```

정리한 이유:

```text
잔여좌석이 1석과 2석일 때의 차이는 크지만,
31석과 32석일 때의 차이는 작기 때문이다.
```

그래서 잔여좌석을 단순 구간으로 자르지 않고, 좌석이 적을수록 더 민감하게 반영되도록 보조 점수도 만들었다.

## 2.5 외부요인 데이터 정리

버스 만차는 버스 자체 정보만으로 결정되지 않는다.

그래서 아래 외부요인도 함께 정리했다.

```text
날씨
기온
강수량
습도
풍속
공휴일 여부
행사 수
근처 행사 여부
```

정리한 이유:

```text
비가 오거나 공휴일이거나 주변에 행사가 있으면 버스 이용 패턴이 달라질 수 있기 때문이다.
```

다만 외부요인은 억지로 다 넣지 않았다.

현재 데이터가 충분히 쌓이지 않은 대기질, 교통량 같은 항목은 모델에서 제외했다.

```text
데이터가 부족한 정보를 억지로 넣으면
모델이 실제와 다른 잘못된 규칙을 배울 수 있기 때문이다.
```

# 3. 알고리즘 개념

## 3.1 사람이 직접 점수를 더하지 않았다

처음에는 다음처럼 점수를 직접 더하는 방식도 생각할 수 있다.

```text
잔여좌석 적음 +50점
비 오는 날 +10점
공휴일 +5점
퇴근 시간 +20점
```

하지만 이 방식은 문제가 있다.

```text
왜 비 오는 날이 꼭 +10점인지 설명하기 어렵다.
노선마다 영향이 다를 수 있다.
정류소마다 영향이 다를 수 있다.
시간대마다 영향이 다를 수 있다.
```

그래서 이 프로젝트에서는 사람이 임의로 가중치를 정하지 않았다.

대신 과거 데이터를 바탕으로 모델이 직접 패턴을 학습하게 했다.

## 3.2 가중치는 어떻게 잡았는가

이 프로젝트에서 말하는 가중치는 두 종류로 구분해야 한다.

```text
1. feature 가중치
2. 학습 데이터 가중치
```

## 3.2.1 feature 가중치

feature 가중치는 사람이 직접 정하지 않았다.

즉 아래처럼 임의 점수를 넣지 않았다.

```text
잔여좌석 0석 = +50점
비 오는 날 = +10점
퇴근 시간 = +20점
공휴일 = +5점
```

이렇게 하지 않은 이유:

```text
노선마다 영향이 다르고,
정류소마다 영향이 다르고,
시간대마다 영향이 다르고,
날씨와 시간대가 함께 작용할 수 있기 때문이다.
```

예를 들어 비가 온다고 항상 만차위험이 높아지는 것은 아니다.  
어떤 노선은 퇴근 시간에 비가 올 때 영향이 클 수 있고, 어떤 노선은 정류소 위치가 더 중요할 수 있다.

그래서 feature별 점수는 사람이 정하지 않고, 모델이 과거 데이터에서 직접 학습하게 했다.

## 3.2.2 학습 데이터 가중치

사람이 직접 정한 가중치는 feature 점수가 아니라, 학습 데이터의 불균형을 보정하기 위한 가중치다.

현재 학습 데이터는 만차보다 비만차가 훨씬 많다.

학습 데이터 기준:

| 구분 | 건수 | 비율 |
| --- | ---: | ---: |
| 전체 학습 row | 66,816건 | 100% |
| 만차 row | 8,145건 | 12.19% |
| 비만차 row | 58,671건 | 87.81% |

이 상태에서 아무 보정 없이 학습하면 모델이 “대부분 안 만차”라고만 예측해도 어느 정도 맞는 것처럼 보일 수 있다.

그래서 학습할 때 만차 row의 중요도를 더 크게 주었다.

적용한 학습 가중치:

| 구분 | 학습 가중치 |
| --- | ---: |
| 만차 row | 4.1017 |
| 비만차 row | 0.5694 |

계산 방식:

```text
만차 가중치 = 전체 학습 row 수 / (2 * 만차 row 수)
비만차 가중치 = 전체 학습 row 수 / (2 * 비만차 row 수)
```

실제 계산:

```text
만차 가중치 = 66,816 / (2 * 8,145) = 4.1017
비만차 가중치 = 66,816 / (2 * 58,671) = 0.5694
```

이렇게 한 이유:

```text
만차 데이터는 적지만 서비스에서 매우 중요한 상황이다.
따라서 학습할 때 만차 사례를 더 가볍게 넘기지 않도록 보정했다.
```

즉 “날씨는 몇 점, 공휴일은 몇 점”처럼 임의 가중치를 준 것이 아니라, 정답 데이터의 불균형을 보정하기 위한 가중치만 적용했다.

## 3.3 모델이 실제로 중요하게 본 정보

LightGBM은 학습이 끝난 뒤 어떤 정보를 자주 사용했는지 중요도를 확인할 수 있다.

아래 값은 사람이 정한 점수가 아니라, 모델이 학습 과정에서 많이 사용한 정보다.

| 순위 | 정보 | 중요도 |
| ---: | --- | ---: |
| 1 | 정류소 순번 | 2,083 |
| 2 | 잔여좌석 수 | 702 |
| 3 | 좌석 부족 점수 | 635 |
| 4 | 기온 | 341 |
| 5 | 시간대 순환값 | 324 |
| 6 | 시간대 순환값 | 293 |
| 7 | 풍속 | 280 |
| 8 | 추정 좌석 수 | 246 |
| 9 | 습도 | 241 |
| 10 | 5005번 노선 여부 | 196 |

해석:

```text
모델은 정류소 순번, 잔여좌석, 좌석 부족 정도를 가장 중요하게 봤다.
그 다음으로 시간대와 날씨 정보도 함께 사용했다.
```

주의할 점:

```text
중요도 2,083은 만차확률에 2,083점을 더한다는 뜻이 아니다.
모델이 예측 규칙을 만들 때 해당 정보를 많이 사용했다는 뜻이다.
```

## 3.4 알고리즘이 하는 일

알고리즘은 과거 데이터를 보면서 이런 관계를 찾는다.

```text
어떤 노선이 어느 시간대에 자주 만차가 되는가
어떤 정류소 근처에서 잔여좌석이 급격히 줄어드는가
공휴일에는 평일과 다른 패턴이 나타나는가
비가 오는 날에는 특정 시간대 만차가 늘어나는가
잔여좌석이 몇 석 이하일 때 다음 정류소에서 만차가 되는가
```

즉 모델은 단순히 현재 좌석만 보는 것이 아니라 여러 조건의 조합을 함께 본다.

## 3.5 최종 예측값

모델이 최종적으로 내보내는 값은 다음 정류소 기준 만차확률이다.

예:

```text
만차확률 92.81%
```

의미:

```text
현재 입력 조건과 비슷한 상황을 종합했을 때,
다음 정류소 기준 만차일 가능성이 높다는 뜻이다.
```

이 값은 사람이 정한 공식으로 계산한 것이 아니라, 학습된 모델이 계산한 확률이다.

# 4. 모델 생성

## 4.1 어떤 모델을 사용했는가

최종 모델은 LightGBM이라는 예측 모델을 사용했다.

쉽게 말하면 LightGBM은 표 형태의 데이터에서 패턴을 잘 찾는 모델이다.

이 프로젝트의 데이터는 이미지나 음성이 아니라 아래처럼 표 형태다.

```text
노선 | 정류소 | 시간 | 잔여좌석 | 혼잡도 | 날씨 | 공휴일 | 만차 여부
```

그래서 표 데이터에 강한 LightGBM을 사용했다.

## 4.2 왜 LightGBM을 사용했는가

이 프로젝트는 표 형태의 공공데이터를 다룬다.

```text
노선
정류소 순번
시간대
요일
잔여좌석
혼잡도
날씨
행사
```

이런 데이터에서는 단순한 공식보다 여러 조건의 조합을 잘 찾는 모델이 필요하다.

LightGBM을 선택한 이유:

| 이유 | 설명 |
| --- | --- |
| 표 데이터에 강함 | 버스 데이터처럼 행과 열로 된 데이터에서 성능이 좋다. |
| 조건 조합을 잘 찾음 | “5001-1B + 신논현역 + 평일 18시 + 잔여좌석 적음” 같은 조합을 학습할 수 있다. |
| 비선형 관계를 학습함 | 잔여좌석 1석과 2석의 차이, 31석과 32석의 차이를 다르게 볼 수 있다. |
| 학습과 예측이 빠름 | 서비스에서 실시간에 가깝게 확률을 계산하기 좋다. |
| 중요도 확인 가능 | 어떤 정보가 예측에 많이 쓰였는지 설명할 수 있다. |

다른 모델과 비교했을 때의 차이:

| 모델 | 장점 | 한계 |
| --- | --- | --- |
| 과거 만차율 기준 모델 | 이해하기 쉽고 단순하다. | 노선, 시간, 날씨, 정류소 조건이 복잡하게 섞이는 상황을 잘 못 잡는다. |
| Logistic Regression | 해석이 쉽고 빠르다. | 각 정보의 영향이 거의 직선적으로 더해진다고 가정해서 복잡한 조건 조합에 약하다. |
| LightGBM | 복잡한 조건 조합과 비선형 패턴을 잘 학습한다. | 단순 공식보다 내부 구조 설명이 어렵다. |

따라서 최종 모델은 LightGBM으로 정했다.

```text
이유:
버스 만차는 단순히 좌석 수 하나로 결정되지 않고,
노선, 정류소 위치, 시간대, 공휴일, 날씨가 함께 작용하기 때문이다.
```

## 4.3 모델이 학습한 문제

모델은 다음 질문에 답하도록 학습했다.

```text
현재 이 차량이 다음 정류소에 도착했을 때 만차가 되는가?
```

정답 데이터는 과거 수집 데이터에서 만들었다.

```text
다음 정류소에서 잔여좌석이 0석이면 만차
다음 정류소에서 잔여좌석이 남아 있으면 비만차
```

이렇게 과거 데이터를 문제와 정답 형태로 만들어 모델에 학습시켰다.

## 4.4 모델에 들어가는 주요 정보

모델에 들어가는 정보는 크게 여섯 묶음이다.

| 정보 묶음 | 의미 |
| --- | --- |
| 버스 상태 | 현재 잔여좌석, 혼잡도, 만차 여부 |
| 노선 | 5005, 5001-1B 같은 노선 정보 |
| 정류소 위치 | 노선 안에서 몇 번째 정류소인지 |
| 시간 | 시간대, 요일, 주말, 공휴일 |
| 날씨 | 기온, 비, 습도, 바람 등 |
| 행사 | 해당 날짜의 행사 영향 |

이 정보를 함께 넣은 이유:

```text
만차는 하나의 원인으로 결정되지 않고,
노선, 시간, 위치, 날씨 같은 조건이 함께 작용하기 때문이다.
```

## 4.5 실시간 예측과 시간표 기반 예측

이 프로젝트에는 예측 방식이 두 가지 있다.

### 실시간 예측

실시간 버스 정보가 있을 때 사용한다.

```text
실제 차량의 현재 잔여좌석
실제 차량의 현재 혼잡도
실제 차량의 현재 위치
```

를 사용하기 때문에 가장 신뢰도가 높다.

### 시간표/과거이력 기반 예측

실시간 버스 정보가 없을 때 사용한다.

예를 들어 사용자가 미래 시간이나 아직 API에 잡히지 않은 차량을 조회하면, 실제 차량 row가 없을 수 있다.

이 경우에는 아래 방식으로 예측한다.

```text
입력한 날짜와 시간 확인
평일/주말/공휴일 여부 확인
그 시간대에 운행하는 노선인지 확인
과거에 같은 노선, 같은 정류소, 비슷한 시간대에서 좌석이 어땠는지 확인
그 값을 임시 입력값으로 만들어 같은 모델에 넣음
```

중요한 점:

```text
실시간 예측과 시간표 기반 예측은 서로 다른 모델이 아니다.
같은 모델을 사용하되, 입력 데이터가 다르다.
```

# 5. 평가지표

## 5.1 왜 성능 검증이 필요한가

모델이 그럴듯한 숫자를 내보내는 것과 실제로 잘 맞추는 것은 다르다.

그래서 과거 데이터 일부를 일부러 학습에 쓰지 않고 남겨둔 뒤, 모델이 그 데이터를 잘 맞추는지 확인했다.

```text
앞쪽 기간 데이터 -> 학습용
뒤쪽 기간 데이터 -> 검증용
```

이렇게 한 이유:

```text
실제 서비스에서는 과거 데이터로 미래를 예측해야 하기 때문이다.
```

## 5.2 사용한 주요 평가지표

| 지표 | 쉬운 의미 |
| --- | --- |
| Accuracy | 전체 예측 중 맞춘 비율 |
| Precision | 만차라고 경고한 것 중 실제 만차였던 비율 |
| Recall | 실제 만차 중 모델이 잡아낸 비율 |
| F1 | Precision과 Recall을 함께 본 점수 |
| ROC-AUC | 만차와 비만차를 구분하는 전체 능력 |
| Brier score | 확률 예측이 실제 결과와 얼마나 가까운지 |
| Top 10% Precision | 위험도가 가장 높은 10% 예측 중 실제 만차였던 비율 |

## 5.3 왜 Accuracy만 보면 안 되는가

버스 데이터에서는 만차가 아닌 경우가 더 많다.

만약 전체의 대부분이 비만차라면, 모델이 계속 “비만차”라고만 해도 정확도가 높아 보일 수 있다.

그래서 Accuracy 하나만 보지 않고, Precision과 Recall을 함께 봤다.

중요하게 본 기준:

```text
만차라고 경고했을 때 실제로 만차인 비율
실제 만차 상황을 얼마나 놓치지 않는지
확률값이 실제 결과와 얼마나 비슷한지
```

## 5.4 현재 검증 결과 해석

실시간 차량 정보가 있는 경우에는 모델 성능이 높게 나왔다.

```text
실제 잔여좌석과 혼잡도를 알고 있기 때문에
다음 정류소 만차 여부를 비교적 잘 예측할 수 있다.
```

반면 실시간 차량 정보가 없고 시간표/과거이력만 사용하는 경우에는 성능이 낮아진다.

```text
실제 차량 상태를 모르는 상태에서
과거 비슷한 상황을 바탕으로 추정하기 때문이다.
```

그래서 서비스에서는 두 결과를 다르게 표시해야 한다.

```text
실시간 기반 예측: 신뢰도 높음
시간표/과거이력 기반 예측: 참고용, 과거 표본 수와 함께 표시
```

## 5.5 다른 모델과의 정량 비교

같은 검증 데이터에서 세 가지 방식을 비교했다.

비교 대상:

```text
1. 과거 만차율 기준 모델
2. Logistic Regression 모델
3. LightGBM 모델
```

검증 조건:

```text
학습 row: 66,816건
검증 row: 16,705건
검증 방식: 시간순 분리
목표: 다음 정류소 기준 만차 여부 예측
```

정량 비교 결과:

| 모델 | Accuracy | Precision | Recall | F1 | ROC-AUC | Brier score | Top 10% Precision |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 과거 만차율 기준 | 86.27% | 54.34% | 36.52% | 43.68% | 83.76% | 0.0994 | 53.75% |
| Logistic Regression | 96.81% | 89.15% | 89.18% | 89.16% | 96.16% | 0.0500 | 91.74% |
| LightGBM | 97.71% | 97.79% | 86.42% | 91.75% | 97.67% | 0.0451 | 98.92% |

해석:

```text
LightGBM은 과거 만차율 기준 모델보다 F1이 48.07%p 높다.
LightGBM은 Logistic Regression보다 F1이 2.59%p 높다.
LightGBM은 Logistic Regression보다 Precision이 8.64%p 높다.
LightGBM은 Logistic Regression보다 Top 10% Precision이 7.18%p 높다.
LightGBM은 Logistic Regression보다 Brier score가 0.0049 낮다.
```

주의:

```text
Brier score는 낮을수록 좋다.
나머지 Accuracy, Precision, Recall, F1, ROC-AUC, Top 10% Precision은 높을수록 좋다.
```

LightGBM의 Recall은 Logistic Regression보다 낮다.

```text
Logistic Regression Recall: 89.18%
LightGBM Recall: 86.42%
차이: -2.76%p
```

하지만 최종 모델은 LightGBM을 선택했다.

선택 이유:

```text
만차라고 경고했을 때 실제 만차일 확률이 더 높다.
위험도 상위 10%를 뽑았을 때 실제 만차 비율이 더 높다.
전체 구분력인 ROC-AUC도 더 높다.
확률 오차인 Brier score도 더 낮다.
```

서비스 관점에서는 잘못된 만차 경고가 너무 많으면 사용자가 서비스를 믿지 않게 된다.  
따라서 초기 모델은 Recall을 조금 양보하더라도 Precision과 Top 10% Precision이 높은 LightGBM을 선택했다.

## 5.6 threshold는 왜 0.9로 잡았는가

모델은 0부터 1 사이의 확률을 낸다.

```text
0.0 = 만차 가능성 거의 낮음
1.0 = 만차 가능성 매우 높음
```

하지만 서비스에서는 어느 정도 이상을 “위험”으로 볼지 기준이 필요하다.  
현재 기준은 0.9다.

```text
fullSeatProbability >= 0.9
  -> high risk
```

LightGBM 기준 threshold별 성능:

| Threshold | 경고 비율 | Precision | Recall | F1 |
| ---: | ---: | ---: | ---: | ---: |
| 0.8 | 13.86% | 94.08% | 88.57% | 91.24% |
| 0.9 | 13.01% | 97.79% | 86.42% | 91.75% |
| 0.95 | 12.68% | 98.21% | 84.63% | 90.91% |

0.9를 선택한 이유:

```text
0.8보다 잘못된 경고가 적다.
0.95보다 실제 만차를 덜 놓친다.
Precision과 Recall의 균형이 좋다.
F1이 세 기준 중 가장 높다.
```

즉 0.9는 “너무 자주 경고하지 않으면서도 실제 만차를 충분히 잡는 기준”으로 선택했다.


# 8. 한계와 보완 방향

현재 모델은 실제 수집된 데이터 범위 안에서 학습했다.

따라서 데이터가 더 많이 쌓이면 더 좋아질 수 있다.

앞으로 보완할 점:

```text
더 긴 기간의 실시간 버스 데이터 수집
노선별 출근/퇴근 패턴 추가 검증
정류소별 승하차 수요 데이터 확보
대기질과 교통정체 데이터 추가
시간표 기반 예측의 표본 수 확대
```

중요한 한계:

```text
실시간 차량 정보가 없을 때의 예측은 실제 차량 관측이 아니라 과거 패턴 기반 추정이다.
따라서 이 경우에는 만차확률과 함께 예측 방식과 참조 이력 수를 반드시 함께 보여줘야 한다.
```
