# 용인 5000계열 노선 데이터 전처리 설계

대상 노선:

```text
5000
5001A
5001B
5001-1A
5001-1B
5003A
5003B
5005
```

목표:

```text
각 노선의 전체 경유정류소를 기준으로
요일, 시간대, 정류소 순번, 차량, 잔여좌석, 혼잡도, 로그 기반 좌석 위험 스코어를
모델 학습용 feature 테이블로 정리한다.
```

# 1. 이번 전처리에서 실제로 쓰는 데이터

```text

GbisClient.route_info()
```

```text

routeTypeCd

regionName

endStationName
```

```text

routeTypeCd 기준 기본 좌석 정원 추정
```

저장 테이블:

```text

```

# 2. 전처리 실행 순서

## 0단계: API 키 설정

`.env` 파일에 공공데이터포털 서비스키를 넣어야 함.

```text
PUBLIC_DATA_SERVICE_KEY=...
BUSSEAT_DB_PATH=data/busseat.db
```

---

## 1단계: DB 초기화

```powershell
busseat init-db
```

생성되는 주요 테이블:

```text
target_route
route_station
bus_location_snapshot
preprocessed_location_features
```

---

## 2단계: 대상 노선과 전체 경유정류소 저장

```powershell
busseat prepare-target-routes
```

이 명령은 아래 작업을 수행함.

```text
1. 5000, 5001, 5001-1, 5003, 5005로 노선 검색
2. 검색 결과에서 5000, 5001A/B, 5001-1A/B, 5003A/B, 5005만 필터링
3. 실제 routeId를 target_route에 저장
4. 각 routeId의 전체 경유정류소를 route_station에 저장
```

특정 지역명으로 추가 필터링하고 싶으면:

```powershell
busseat prepare-target-routes --region-keyword 용인
```

단, API 응답의 `regionName`이 비어 있거나 시작/종점명에 용인이 안 들어가는 경우가 있을 수 있어서 기본값은 지역 필터 없음으로 둠.

---

## 3단계: 대상 노선 요약 확인

```powershell
busseat target-summary
```

확인할 것:

```text
routeId가 제대로 잡혔는지
5001A/B 같은 분기 노선이 빠지지 않았는지
노선별 경유정류소 수가 0이 아닌지
```

---

## 4단계: 위치/잔여좌석 스냅샷 수집

```powershell
busseat collect-target-locations
```

이 명령은 `target_route`에 저장된 모든 routeId에 대해 위치정보를 조회하고 저장함.

저장되는 값:

```text
collected_at
route_id
veh_id
plate_no
station_id
station_seq
remain_seat_cnt
crowded
route_type_cd
low_plate
state_cd
tagless_cd
raw_json
```

수집 주기 추천:

```text
MVP: 1분 간격
공모전 데모용: 출퇴근 시간대 30초~1분 간격
```

PowerShell에서 반복 수집 예시:

```powershell
while ($true) {
  busseat collect-target-locations
  Start-Sleep -Seconds 60
}
```

---

## 5단계: 버스 feature 전처리

```powershell
busseat preprocess-location-features
```

CSV까지 내보내려면:

```powershell
busseat preprocess-location-features --export-csv data/location_features.csv
```

---

## 6단계: 스냅샷 전체 기간의 시간 단위 기상자료 수집

버스 스냅샷에 들어있는 전체 날짜 범위를 자동으로 추론해서 기상청 ASOS 시간자료를 받음.

```powershell
busseat fetch-weather-hourly --station-id 119
```

직접 날짜 범위를 지정하려면:

```powershell
busseat fetch-weather-hourly --station-id 119 --start-date 2026-05-01 --end-date 2026-05-31
```

현재 기본 `119`는 수원 ASOS 지점 기준임.  
용인 전체를 대표하기에 완벽하지는 않으므로, 실제 제출 전에는 대상 정류소 좌표 기준 가장 가까운 ASOS/AWS 지점으로 교체하는 게 좋음.

---

## 7단계: 대기질/교통/공휴일/행사 외부요인 적재

아래 데이터는 제공 API의 형태가 서로 다르므로, 우선 표준 CSV로 받아서 적재할 수 있게 해둠.

```powershell
busseat import-external-csv air data/air_quality_hourly.csv
busseat import-external-csv traffic data/traffic_hourly.csv
busseat import-external-csv holiday data/holiday_daily.csv
busseat import-external-csv event data/event_daily.csv
```

각 CSV의 최소 컬럼:

```text
air:
  observed_at_kst, pm10, pm25, o3, khai, air_quality_grade

traffic:
  observed_at_kst, avg_speed, traffic_volume, delay_time, congestion_level

holiday:
  service_date, is_holiday, holiday_name

event:
  service_date, area_key, event_count, event_nearby_count
```

---

## 8단계: 최종 학습용 시간 단위 feature 생성

버스 feature와 외부요인을 시간 단위로 결합함.

```powershell
busseat build-model-hourly-features --export-csv data/model_hourly_features.csv
```

결합 기준:

```text
버스 스냅샷 시각 -> KST 시간 단위로 내림
날씨/대기질/교통 -> 같은 시간 값 우선 사용
같은 시간이 없으면 가장 가까운 시간 값 사용
행사/공휴일 -> 같은 날짜 값 우선 사용
같은 날짜 값이 없으면 가장 가까운 날짜 값 또는 기본값 사용
```

결측 보정 여부:

```text
weather_imputed
air_quality_imputed
traffic_imputed
event_imputed
```

값이 1이면 정확히 같은 시간/날짜가 아니라 가까운 값 또는 기본값으로 채운 것임.

# 3. 전처리 결과 테이블

전처리 결과는 아래 테이블에 저장됨.

```text
preprocessed_location_features
```

## 시간 feature

| 컬럼 | 의미 |
| --- | --- |
| `collected_at_utc` | 원본 수집 시각 UTC |
| `collected_at_kst` | 한국시간 변환 시각 |
| `service_date` | 한국시간 기준 날짜 |
| `day_of_week` | 월=0, 화=1, ..., 일=6 |
| `day_name_ko` | 월/화/수/목/금/토/일 |
| `is_weekend` | 토/일이면 1 |
| `hour` | 한국시간 시 |
| `minute` | 한국시간 분 |
| `time_bucket_10m` | 10분 단위 시간대, 예: `08:00`, `08:10` |
| `time_period` | `morning_peak`, `daytime`, `evening_peak` 등 |

## 노선 feature

| 컬럼 | 의미 |
| --- | --- |
| `route_id` | GBIS routeId |
| `route_name` | API 노선명 |
| `canonical_route_name` | 정규화된 노선명, 예: `5001-1A` |
| `route_type_cd` | 노선 유형 코드 |
| `route_type_name` | 노선 유형명 |

## 차량 feature

| 컬럼 | 의미 |
| --- | --- |
| `veh_id` | 차량 ID |
| `plate_no` | 차량 번호 |
| `low_plate` | 저상버스 여부 코드 |
| `state_cd` | 운행 상태 코드 |
| `tagless_cd` | 태그리스 관련 코드 |

## 정류소 feature

| 컬럼 | 의미 |
| --- | --- |
| `station_id` | 정류소 ID |
| `station_seq` | 노선 내 정류소 순번 |
| `station_name` | 정류소명 |
| `mobile_no` | 정류소 ARS 번호 |
| `x` | 정류소 경도 |
| `y` | 정류소 위도 |

## 잔여좌석/혼잡 feature

| 컬럼 | 의미 |
| --- | --- |
| `remain_seat_cnt` | 잔여좌석 수 |
| `crowded` | 혼잡도 코드 |
| `crowded_label` | 여유/보통/혼잡/매우혼잡 |
| `estimated_capacity` | 추정 좌석 정원 |
| `seat_scarcity_score` | 로그 기반 잔여좌석 위험 스코어 |
| `is_no_seat` | 잔여좌석 0석이면 1 |
| `is_low_seat_2` | 잔여좌석 0~2석이면 1 |
| `is_low_seat_5` | 잔여좌석 0~5석이면 1 |

# 4. 요일/시간대 기준

## 요일

Python 기준으로 저장함.

```text
월요일 = 0
화요일 = 1
수요일 = 2
목요일 = 3
금요일 = 4
토요일 = 5
일요일 = 6
```

`day_name_ko`는 사람이 보기 쉽게 따로 저장함.

```text
월, 화, 수, 목, 금, 토, 일
```

## 시간대

현재 MVP 기준:

```text
05:00~06:59 -> early_morning
07:00~09:59 -> morning_peak
10:00~16:59 -> daytime
17:00~19:59 -> evening_peak
20:00~23:59 -> night
00:00~04:59 -> late_night
```

이 기준은 초기값임.  
데이터가 쌓이면 실제 만차율이 높은 시간대를 보고 조정 가능함.

# 5. 잔여좌석 스코어

잔여좌석 수는 이산 점수로 자르지 않음.

사용 공식:

```text
seatScarcityScore
  = 100 * (1 - (ln(remainSeatCnt + 1) / ln(seatCapacity + 1)) ^ 2.15)
```

45석 기준:

```text
0석  -> 100점
1석  -> 97점
2석  -> 93점
3석  -> 89점
4석  -> 84점
5석  -> 80점
10석 -> 63점
20석 -> 39점
45석 -> 0점
```

이 값은 확률이 아니라 **화면용 위험 스코어**임.

# 6. 모델 학습 전처리 기준

처음 만들 학습 데이터 단위:

```text
1 row = 특정 시각에 특정 노선의 특정 차량이 특정 정류소 순번에 있을 때의 상태
```

즉:

```text
collected_at_kst
route_id
veh_id
station_seq
remain_seat_cnt
```

조합이 하나의 관측치가 됨.

## 1차 feature

```text
day_of_week
is_weekend
hour
time_bucket_10m
time_period
canonical_route_name
route_type_cd
station_seq
station_id
remain_seat_cnt
crowded
seat_scarcity_score
is_no_seat
is_low_seat_2
is_low_seat_5
```

## 1차 target 후보

```text
target_no_seat_now
  = is_no_seat

target_low_seat_2_now
  = is_low_seat_2

```


# 7. 외부요인 결측 보정 기준

최종 테이블 `model_hourly_features`는 모델 학습 편의를 위해 NULL이 없도록 만듦.

보정 우선순위:

```text
1. 같은 시간/날짜의 값
2. 가장 가까운 시간/날짜의 값
3. 데이터가 아예 없으면 안전한 기본값
```

기본값:

```text
날씨: temperature=0, precipitation=0, humidity=0, wind_speed=0, weather_text=unknown
대기질: pm10=0, pm25=0, o3=0, khai=0, air_quality_grade=unknown
교통: avg_speed=0, traffic_volume=0, delay_time=0, congestion_level=0
행사: event_count=0, event_nearby_count=0
공휴일: is_holiday=0
```

주의:

```text
기본값은 “영향 없음”이 아니라 “데이터 없음”을 모델이 알 수 있게 imputed 플래그와 같이 써야 함.
```

# 8. 지금 한계

현재 전처리는 `경기도_버스위치정보 조회` 중심임.

장점:

```text
노선별 호출이라 모든 정류소 순번을 효율적으로 커버 가능
차량별 잔여좌석 변화 추적 가능
```

한계:

```text
특정 정류소에 곧 도착할 1번째/2번째 차량 기준 예측은 도착정보 API와 결합해야 함
정류소에서 실제 몇 명이 탔는지는 직접 제공되지 않음
정류소별 승하차 인원 통계와 결합해야 expected_boardings_at_stop 추정 가능
```
