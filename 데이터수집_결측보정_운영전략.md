# 데이터 수집·결측 보정·운영 전략 정리

이 문서는 나중에 종합 보고서 작성할 때 그대로 가져다 쓰기 위한 정리 문서임.

핵심 요약:

```text
버스 위치/도착 데이터는 실시간 API 중심이라 과거 전체 데이터를 한 번에 받을 수 없다.
따라서 우리가 직접 주기 수집해서 자체 이력 DB를 만들어야 한다.

날씨/공휴일처럼 과거 조회가 가능한 데이터는 기간 단위로 보강한다.
대기질/교통처럼 실시간 성격이 강한 데이터는 버스 데이터와 동시에 누적 수집하거나 CSV로 보완한다.

결측값은 무조건 0으로 덮지 않고,
같은 시간 -> 가까운 시간 -> 기본값 순서로 채우며,
보정 여부를 *_imputed 컬럼에 반드시 남긴다.
```

---

# 1. 데이터별 수집 가능성 정리

## 1. 버스 위치/도착 데이터

사용 데이터:

```text
경기도_버스위치정보 조회
경기도_버스도착정보 조회
```

현재 판단:

```text
실시간 API 중심
과거 전체 차량 위치/잔여좌석 이력을 한 번에 받는 공식 API는 확인하지 못함
```

의미:

```text
“과거 전체 데이터 다운로드” 방식으로는 모델 학습 데이터를 만들기 어렵다.
지금부터 대상 노선을 주기적으로 호출해 자체 DB에 누적해야 한다.
```

운영 방식:

```text
버스위치정보: 30초~1분 주기
버스도착정보: 30초~1분 주기
수집 대상: 5000, 5001A/B, 5001-1A/B, 5003A/B, 5005
수집 범위: 각 노선 전체 경유정류소, 전체 운행 차량
```

현재 저장 테이블:

```text
bus_location_snapshot
bus_arrival_snapshot
```


# 3. 결측 보정 원칙

## 기본 원칙

```text
NULL을 그대로 두지 않는다.
하지만 그냥 0으로 덮지 않는다.
어떤 값이 보정됐는지 imputed 플래그로 남긴다.
```

---

## 시간 단위 데이터 보정

대상:

```text
날씨
대기질
교통
```

보정 순서:

```text
1. 같은 observed_hour_kst 값 사용
2. 같은 시간이 없으면 가장 가까운 observed_at_kst 값 사용
3. 해당 외부요인 데이터가 하나도 없으면 기본값 사용
```

예:

```text
버스 스냅샷 시간: 2026-05-21 08:10
시간 단위 기준: 2026-05-21 08:00

날씨 08:00 있음 -> 그대로 사용, weather_imputed=0
대기질 08:00 없음, 07:00 있음 -> 07:00 사용, air_quality_imputed=1
교통 데이터 없음 -> 기본값 사용, traffic_imputed=1
```

---

## 날짜 단위 데이터 보정

대상:

```text
공휴일
행사
```

보정 순서:

```text
1. 같은 service_date 값 사용
2. 같은 날짜가 없으면 가장 가까운 날짜 값 사용
3. 데이터가 하나도 없으면 기본값 사용
```

# 4. 결측 보정이 많이 쌓이는 경우

결측 보정이 비정상적으로 많아지는 상황:

```text
1. 버스 스냅샷은 많이 쌓였는데 외부요인 수집을 안 한 경우
2. 외부요인 수집 주기가 너무 긴 경우
3. 대기질/교통처럼 실시간 API만 있고 과거 이력이 부족한 경우
4. 정류소와 측정소/도로구간 매칭이 아직 안 된 경우
```

예시:

```text
버스 데이터: 1분마다 10,000건
대기질 데이터: 하루 1건

결과:
대부분의 버스 행에서 air_quality_imputed=1
```

이 경우 대기질 feature는 모델에서 제외하거나, 수집 주기를 개선해야 함.

# 5. Feature 신뢰도 기준

모델 학습 전에 아래 기준으로 feature 사용 여부를 판단함.

## 권장 기준

```text
weather_imputed 비율 <= 30%
  날씨 feature 사용 가능

air_quality_imputed 비율 <= 50%
  대기질 feature 사용 가능

traffic_imputed 비율 <= 50%
  교통 feature 사용 가능

event_imputed 비율은 날짜 단위라 높아도 별도 검토
```

## 제외 기준

```text
특정 feature의 imputed 비율이 너무 높으면 모델 학습에서 제외한다.
```

예:

```text
traffic_imputed = 90%

판단:
교통 feature는 현재 데이터 품질로는 모델에 넣지 않는다.
보고서에는 “교통소통 데이터는 수집 기간 부족으로 추후 고도화 항목”이라고 쓴다.
```

# 6. 보고서에 쓸 수 있는 문장

## 버스 데이터 한계

```text
경기도 버스 위치/도착 API는 실시간 운행정보 중심으로 제공되므로,
과거 전체 잔여좌석 이력을 일괄 다운로드하는 방식은 확인되지 않았다.
따라서 본 프로젝트는 대상 노선을 주기적으로 호출하여 자체 시계열 이력 DB를 구축하는 방식으로 설계하였다.
```

## 외부요인 결합

```text
기상 데이터는 기상청 ASOS 시간자료를 이용하여 시간 단위로 결합하였다.
대기질, 교통소통, 행사, 공휴일 데이터는 제공 주기와 공간 단위가 서로 다르기 때문에
시간 단위 또는 날짜 단위 표준 테이블로 정규화한 뒤 모델 feature에 결합하였다.
```

## 결측 보정

```text
외부요인 데이터의 결측값은 동일 시간 값을 우선 사용하고,
존재하지 않는 경우 가장 가까운 시간 또는 날짜의 값으로 보정하였다.
보정 여부는 weather_imputed, air_quality_imputed, traffic_imputed, event_imputed 컬럼에 기록하여
모델 학습 시 feature 신뢰도를 평가할 수 있도록 하였다.
```

## feature 제외 기준

```text
보정값 비율이 높은 feature는 모델에 무조건 포함하지 않고,
imputed 비율을 기준으로 feature 사용 여부를 결정하였다.
이를 통해 결측 보정이 모델의 주요 설명 변수가 되는 문제를 방지하였다.
```
